- **실험/성공조건**: 빌드 성공 → ✅ `Build succeeded`
- **커밋**: `a7fca62`

### ✅ Task 2 — AssetIdGenerator
- **무엇을**: 섹션명+순번을 파일명 안전한 ID로 ("표지", 1 → "표지_01")
- **왜**: 쪼갠 에셋마다 충돌 없는 고유 파일명이 필요. 분할계획(T3)의 재료
- **실험/성공조건**: xUnit 3개 PASS (2자리 패딩, 금지문자·괄호→`_`) → ✅ 22/22 GREEN
- **커밋**: `9993d64`

### ✅ Task 3 — IngestPlanner
- **무엇을**: 섹션 목록 → 슬라이드 1장=에셋 1개로 펼친 분할 계획(`AssetSplitItem` 리스트)
- **왜**: "어느 슬라이드를 어떤 카테고리·ID·파일명으로 쪼갤지"를 COM 작업 전에 순수 계산
- **실험/성공조건**: xUnit 2개 PASS (카테고리별 순번 1부터, 빈 섹션→빈 결과) → ✅ 24/24 GREEN
- **커밋**: `f2c7db9`

> 🔹 순수 로직 3개(T1~3) 완료 — 단위테스트 GREEN. 이제 실제 PowerPoint를 다루는 COM 어댑터(T4~6).

### ✅ Task 4 — SectionReader (COM)
- **무엇을**: 열린 Presentation의 `SectionProperties` → `SectionInfo` 리스트 (섹션명·첫슬라이드·슬라이드수)
- **왜**: T3 계획의 입력인 "실제 pptx의 섹션 구조"를 읽어오는 손
- **실험/성공조건**: 빌드 성공(Interop API 존재 확인) → ✅ Build succeeded. 동작은 T7에서 검증
- **커밋**: `c41e1bf`

### ✅ Task 5 — SlideSplitter (COM)
- **무엇을**: 소스 pptx의 슬라이드 1장을 복사 → 새 빈 프레젠테이션에 붙여 단일 슬라이드 pptx로 저장
- **왜**: "슬라이드 1장 = 에셋 1개"를 실제 파일로 떼어내는 손
- **실험/성공조건**: 빌드 성공 → ✅ Build succeeded. (⚠️ T7에서 분할 pptx가 1슬라이드만인지 확인)
- **커밋**: `2cf8354`

### ✅ Task 6 — SlideImageRenderer (COM)
- **무엇을**: 슬라이드를 PNG로 렌더, 긴 변을 768px(`LlmImageLongEdgePx`)에 맞춰 비율 유지 다운스케일
- **왜**: LLM이 "보고 이해"할 썸네일. 768px = 토큰 효율 전역 기본
- **실험/성공조건**: 빌드 성공 → ✅ Build succeeded. 동작은 T7에서 검증
- **커밋**: `a161876`

### 🔵 Task 7 — IngestRunner (오케스트레이터, 수동검증 대기)
- **무엇을**: 묶음 pptx 열기 → 섹션읽기(T4) → 계획(T3) → 각 항목 split(T5)+render(T6) → 닫기. 2~6을 엮는 지휘자
- **왜**: 부품들을 하나의 "묶음 pptx → 낱장 에셋들" 파이프라인으로 완성
